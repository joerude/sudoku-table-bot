# Календарные сезоны + панель инициатив

Дата: 2026-07-20. Статус: одобрено (интервью в чате).

## Проблема

Сезон закрывается только по достижению target (100 очков). При падении темпа
(38 сезонных игр/нед → 5) сезон растягивается на месяцы: лидер за 11 дней
сезона 5 набрал 24 из 100. Прогресс сезона — заложник самого занятого игрока.
Плюс нет видимости, кто зовёт играть, а кто пассивен.

## Решения (из интервью)

- Сезон заканчивается **по календарю: ~месяц** (выбран календарный месяц,
  не 2 недели, не 30-дней-от-старта, не настраиваемый срок).
- Target 100 остаётся **досрочным финалом**: что наступит раньше.
- Пустой месяц (0 сезонных игр) → сезон **продлевается** на месяц, без поста.
- Статы инициатив — **только из существующих данных** (`created_by`),
  без трекинга несостоявшихся зовов.

## 1. Календарные сезоны

### Модель

- Новая колонка `seasons.deadline TEXT` (UTC datetime, формат как `started_at`).
- При создании сезона (все три пути: `ActiveSeason` bootstrap, `CreateSeason`,
  `CloseSeason` → следующий): deadline = конец календарного месяца создания
  по TZ чата; **если до конца месяца < 7 дней — конец следующего месяца**
  (досрочка 28-го не порождает 3-дневный сезон).
- Заполнение существующих сезонов — **ленивое, в тикере**, а не в `migrate()`:
  слой БД не знает TZ чата, а дедлайн от неё зависит. Активный сезон с
  `deadline IS NULL` получает дедлайн на первой же минуте работы бота
  (сезон 5 → 1 августа 2026 00:00 по Asia/Bishkek). Тот же путь закрывает
  дырку, если запись дедлайна новому сезону сорвалась.

### Закрытие

Минутный тикер `runReminders` получает шаг `closeExpiredSeasons`:
для каждого чата с активным сезоном, когда `now >= deadline`:

- **Сезонных игр ≥ 1** → `CloseSeason(season, standings[0].PlayerID)`
  (тай-брейк уже в `Standings`: очки → победы → имя) + пост `seasonEndText`
  с наградами (`seasonAwards`), как при досрочном финале. Новый сезон
  получает свой deadline по правилу выше.
- **Игр 0** → `deadline += 1 месяц` (конец следующего месяца), тихо.

Проверка `now >= deadline` переживает даунтайм бота (не «граница месяца»).
Досрочный финал по target в `scoreAndCheck` не меняется.

Нумерация сезонов остаётся числовой (`number`), месяцы в названия не тащим.

### Хранилище

- `SeasonDeadline`/`SetSeasonDeadline` либо deadline в структуре `Season` —
  решается в имплементации; расчёт «конец месяца по TZ» — чистая функция
  в `domain` (покрывается тестами: обычный месяц, декабрь→январь, правило
  <7 дней, продление пустого).

## 2. Панель инициатив

Новый таб в `/stats` (`statsTabs` + `statsView`): «🎯 Актив».

На игрока (активные игроки чата):

- инициировано игр/дуэлей за всё время и за последние 30 дней
  (`games.created_by` → `players.tg_id`; завершённые, не удалённые);
- «дней с последней игры» — последнее участие в любой завершённой игре,
  включая дуэли (семантика `CompletedToday`).

Legacy-игры с невалидным `created_by` (222 шт.) не считаются — без сносок.
SQL-метод в `storage` (покрыт тестом), рендер в `messages.go` (покрыт тестом).

## Не делаем

- Трекинг «звал, но не сыграли» (новая таблица событий).
- Настройку длительности сезона.
- Catch-up механику для отстающих.
- Переименование сезонов в месяцы.

## Тесты

- domain: расчёт deadline (границы месяцев, TZ, правило <7 дней, продление).
- storage: round-trip deadline через все геттеры сезона + переживание
  повторного `Open` (schema.sql и `migrate()` бегут заново); выборка инициатив.
- рендер таба — messages_test.
- Telegram-glue (тикер, посты) — без тестов, по конвенции репо.
